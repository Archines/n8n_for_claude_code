# n8n Workflow 開発ルール

ワークフロー設計・実装時に従うべきルール集。

---

## 1. 命名規則

### Workflow名
- 日本語で記載
- 形式: `[カテゴリ] 機能名 - 説明`
- 例: `[Slack] 通知送信 - 日次レポート`

### ノード名
- 処理内容を明確に示す名前を付ける
- デフォルト名（HTTP Request, Set など）は使用しない
- 例: `顧客データ取得`, `Slack通知送信`, `日付フォーマット変換`

### 変数・フィールド名
- キャメルケース（camelCase）
- 意味のある名前を付ける
- 例: `customerName`, `orderDate`, `totalAmount`

### ファイル名
- 形式: `workflow-[カテゴリ]-[機能名].json`
- 例: `workflow-slack-daily-report.json`

## 2. 構造設計

### モジュール化
- 1つのWorkflowは1つの責務に集中
- 複雑な処理は複数Workflowに分割し、Sub-workflowとして呼び出す
- 再利用可能な処理は共通Workflowとして切り出す

### エラーハンドリング
- 全ての外部API呼び出しにError Workflowを設定
- Try/Catchパターンを使用
- エラー発生時は通知（Slack, Email等）を必ず行う

### ノード配置
- 左から右へ処理の流れに沿って配置
- 分岐は上下に展開
- 関連するノードはグループ化

## 3. ノード選択・実装

### Code Node
- **可読性の観点からPythonを使用して実装する**
- Pythonで実装できない場合に限りJavaScriptを利用
  - 例: 他ノードの出力を参照する必要がある場合（Pythonの`_node`は不可）
- コードは適切にコメントを付ける

### API連携ノードの選択
- **APIを利用する場合、専用ノードがあれば必ずそれを使用する**
  - 例: Google Calendar → Google Calendar Node（HTTP Requestではなく）
  - 例: Slack → Slack Node
- **例外**: 専用ノードでは実現できない操作 → HTTP Request Node使用可
  - その場合、なぜ専用ノードを使用しなかったかをStickyNoteで説明

### データベース
- ベクトルDB・RDBは **SupabaseのPostgreSQL** を使用

## 4. セキュリティ

- Credentialsは必ずn8nの認証情報機能を使用（ハードコード禁止）
- 環境ごと（開発/本番）に認証情報を分ける
- 機密情報はログに出力しない
- 個人情報は必要最小限のみ取得
- 外部送信データは事前にマスキング/フィルタリング

## 5. パフォーマンス

- 大量データは適切なバッチサイズで処理（推奨: 100-500件）
- Split In Batchesノードを活用
- Rate Limitを考慮した待機時間を設定
- HTTP Requestにはタイムアウト設定（推奨: 30秒）
- 不要なデータは早めに削除（Setノードで不要フィールドを除外）

## 6. テスト・デバッグ

- 本番環境でのテスト実行は禁止
- テスト用のダミーデータを準備
- テスト用Credentialsを分離
- 本番デプロイ前にデバッグ用ノードを削除または無効化

## 7. ドキュメント

- Workflowの設定画面で目的・トリガー条件・処理概要を記載
- 複雑なロジックにはStickyNoteノードで説明を追加
- 重要な変更は変更履歴として記録
